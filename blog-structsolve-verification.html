How I Validated a Structural Solver Against 52 Textbook Problems

March 2026 8 min read Olaitan Hammed

StructSolve is a structural analysis web application I built for civil and structural engineers. It handles continuous beam analysis using the Slope Deflection Method and general plane frame analysis using the Direct Stiffness Method. But building the solver was only half the challenge — the other half was proving it actually works.

When you're building software that engineers will use for real calculations — calculations that affect the safety of buildings and structures — "it looks right" isn't good enough. You need rigorous, documented verification against known solutions.

The Verification Standard

I chose Aslam Kassimali's "Structural Analysis" textbook as the verification benchmark. It's a widely-used reference in structural engineering education, and every problem has a known, worked-out solution. If my solver could match every answer in the book, it would prove the calculations are trustworthy.

I set up a comprehensive test suite covering two categories:

  • Continuous Beam Problems — Beams with multiple spans, various loading conditions (point loads, distributed loads, moments), fixed and pinned supports, overhangs, and settlement cases
  • General Frame Problems — Portal frames, multi-story frames, frames with sway, inclined members, and combined loading scenarios

Building the Test Suite

The test suite was a Python script (test_kassimali.py) that programmatically ran each problem through the solver and compared the output against the textbook's expected values. For each problem, the test verified:

  1. Support reactions — Vertical and horizontal forces, moments at supports
  2. Member end forces — Shear and moment values at each end of every member
  3. Equilibrium checks — Sum of forces and moments should equal zero (within tolerance)
  4. Diagram data — Shear force and bending moment diagram values at key points

Each comparison used a tolerance of 0.5% — tight enough to catch real errors, but flexible enough to account for rounding differences between the textbook's manual calculations and the solver's computational precision.

The Bugs I Found

Running 52 problems against the solver didn't produce 52 immediate passes. Here's what I discovered and fixed along the way:

Sign Convention Mismatch

One of the first failures came from a sign convention inconsistency in how the frame solver reported shear forces. The internal calculation was correct, but the output was flipping signs for members oriented in certain directions. This is the kind of bug that's invisible if you only test simple cases — it only appears when you test frames with inclined members.

Diagram Data Edge Case

The get_diagram_data method in the beam solver had a bug where the shear force diagram would produce incorrect values at the exact location of a point load. The discontinuity wasn't being handled properly. A targeted fix to the interpolation logic resolved this.

Sway Detection Accuracy

For frames with sway (horizontal displacement), the auto-detection system occasionally miscategorized a non-sway frame as having sway. This led to extra degrees of freedom in the stiffness matrix and slightly different results. Tightening the geometric analysis thresholds fixed the classification.

The Results

52 / 52
All verification checks passed

After fixing the bugs uncovered by the test suite, every single problem matched the textbook's solutions within the defined tolerance. The test suite now runs as part of the development workflow — any future changes to the solver must pass all 52 problems before they're accepted.

Why This Matters

Engineering software without verification is just a calculator that might be wrong.

The difference between a demo project and production-grade engineering software is trust. By validating against a recognized academic standard, StructSolve can be used with confidence for real engineering work — not just homework problems.

This verification process also changed how I approach building any software that handles critical calculations. The approach is straightforward:

  1. Find a recognized reference — textbooks, published papers, industry standards
  2. Build automated tests — not manual spot-checks, but comprehensive programmatic tests
  3. Set tight tolerances — don't let errors hide behind loose thresholds
  4. Test edge cases — the simple cases always work; it's the unusual configurations that reveal bugs
  5. Keep the tests running — verification isn't a one-time event, it's a continuous process

StructSolve is live at structsolve.streamlit.app, and the source code (including the full test suite) is available on GitHub.
